# Frameworks de Referência

Este documento detalha os frameworks teóricos utilizados na skill `discovery-opportunity`.

---

## Opportunity Solution Tree (Teresa Torres)

### O que é

Framework visual que conecta resultados desejados a oportunidades e soluções. Criado por Teresa Torres, autora de "Continuous Discovery Habits".

### Estrutura

```
Resultado Desejado (Outcome)
    │
    ├── Oportunidade 1
    │       ├── Solução A
    │       └── Solução B
    │
    ├── Oportunidade 2
    │       ├── Solução C
    │       └── Solução D
    │
    └── Oportunidade 3
            └── Solução E
```

### Princípios-chave

1. **Outcome over output** — foque no resultado, não na entrega
2. **Oportunidades são problemas ou necessidades** — não são soluções
3. **Múltiplas soluções por oportunidade** — evita apego à primeira ideia
4. **Descoberta contínua** — a árvore evolui com aprendizado

### Quando usar

- Quando há muitas ideias de funcionalidades mas pouca clareza sobre problemas
- Quando o time está pulando direto para soluções
- Para priorizar o que resolver primeiro

### Referências

- Livro: "Continuous Discovery Habits" — Teresa Torres
- Site: producttalk.org

## Jobs to Be Done (JTBD)

### O que é

Teoria que propõe que clientes "contratam" produtos para fazer um "trabalho" específico. Desenvolvida por Clayton Christensen e refinada por Anthony Ulwick.

### A Fórmula JTBD

```
Quando [SITUAÇÃO], 
eu quero [MOTIVAÇÃO], 
para que [RESULTADO ESPERADO].
```

### Exemplos

**Genérico (ruim):**
> "O usuário quer ser mais produtivo"

**Específico (bom):**
> "Quando estou no trânsito voltando de uma visita, quero registrar o que foi discutido por voz, para não esquecer detalhes importantes e não perder tempo digitando depois"

**Genérico (ruim):**
> "O gestor quer relatórios"

**Específico (bom):**
> "Quando chego na reunião de segunda-feira, quero ter os números da semana anterior consolidados automaticamente, para não gastar domingo preparando slides"

### Tipos de Jobs

| Tipo | Descrição | Exemplo |
|------|-----------|---------|
| **Funcional** | Tarefa prática a completar | Enviar relatório para o cliente |
| **Emocional** | Como quer se sentir | Sentir-se no controle da situação |
| **Social** | Como quer ser percebido | Parecer competente para a diretoria |

### Quando usar

- Para ir além de "o que o usuário quer" para "por que ele quer"
- Para descobrir concorrentes não óbvios (outros "contratados" para o mesmo job)
- Para evitar construir funcionalidades que ninguém usa

### Perguntas úteis

- "O que você está tentando realizar quando usa [solução atual]?"
- "Quando foi a última vez que você precisou fazer isso? Me conta o contexto."
- "Se isso não existisse, o que você faria?"
- "O que define 'missão cumprida' para você?"

### Referências

- Livro: "Competing Against Luck" — Clayton Christensen
- Livro: "Jobs to Be Done" — Anthony Ulwick
- Framework: Outcome-Driven Innovation (ODI)

## Combinando os Frameworks

Na prática, usamos JTBD para **descobrir** oportunidades e OST para **organizar e priorizar**.

### Fluxo recomendado

1. **Identificar jobs** — use JTBD para entender o que usuários tentam fazer
2. **Mapear dores** — identifique onde o job atual é difícil/frustrante
3. **Organizar em árvore** — use OST para conectar dores ao resultado desejado
4. **Priorizar** — escolha quais oportunidades atacar primeiro
5. **Só então pensar em soluções** — gere alternativas para as oportunidades priorizadas

### Armadilhas comuns

| Armadilha | Como evitar |
|-----------|-------------|
| Pular para soluções | Pergunte "qual problema isso resolve?" |
| Jobs muito amplos | Peça contexto específico e situação |
| Confundir dor com desejo | "Isso atrapalha hoje ou seria bônus?" |
| Ignorar jobs emocionais/sociais | Pergunte como a pessoa quer se sentir |
